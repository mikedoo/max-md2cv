# 模板系统详细改造计划

## 1. 背景与目标

当前项目的模板系统本质上还是“CSS 模板切换”：

- 模板数据结构只有 `id / name / css`
- 模板默认配置依赖从 CSS 反向解析
- 预览工具栏的配置项是硬编码的
- 模板资源加载、打包和保存都只围绕单个 `.css` 文件展开

这套方案在模板数量少、配置项固定时足够简单，但不适合后续要做的“模板自定义功能”。如果继续沿着当前方向迭代，新增模板或新增模板能力时，通常都要同步修改：

- 后端模板加载逻辑
- 前端 store 类型
- 样式解析逻辑
- 配置 UI
- 预览运行时注入逻辑

这会直接违背两个目标：

1. 新增模板时不修改其他业务代码
2. 后续允许用户基于规则自定义模板

本次改造的目标是把模板系统升级为“声明式模板包”，将模板能力从“切换 CSS 文件”提升为“模板描述 + 样式资源 + 通用配置 schema + 稳定渲染契约”。

## 2. 设计原则

### 2.1 核心原则

- 模板新增零业务代码接入
- 模板配置完全数据驱动
- 预览与导出共用同一套模板运行结果
- 模板能力上限尽量高，但不允许引入不可控执行逻辑
- 保持 DRY，避免模板定义分散在多个硬编码位置
- 保持前后端边界清晰：文件系统、模板包读写仍由 Tauri/Rust 负责

### 2.2 关键取舍

要实现“最大自由”，不能把模板做成任意 JS/Vue 代码插件。那样虽然看似自由，但会带来以下问题：

- 预览与导出一致性难保证
- 模板安全边界消失
- 用户自定义模板功能无法做稳定约束
- 调试和排障成本急剧上升

因此推荐的上限是：

- 核心程序负责 Markdown -> 语义化 HTML
- 模板包负责样式、声明式配置、资源和可选布局规则
- 不允许模板执行任意脚本

这是“高自由且可维护”的边界。

## 3. 当前系统问题诊断

### 3.1 模板模型过窄

当前 `src/stores/resume.ts` 中模板只有：

```ts
interface ResumeTemplate {
  id: string
  name: string
  css: string
}
```

问题：

- 无法承载模板 schema
- 无法承载模板默认值
- 无法承载模板资源清单
- 无法承载模板能力开关
- 无法承载未来用户模板元信息

### 3.2 模板默认值来源不稳定

当前 `src/utils/templateStyle.ts` 通过解析 CSS 选择器和属性，反推出：

- 主题色
- 字体
- 字号
- 行高
- 页边距
- 联系方式模式

问题：

- 解析规则高度脆弱
- 模板能力上限被解析器写死
- 新字段要扩展时必须改 TypeScript
- 某些模板样式写法一旦变化，默认值解析就会失真

### 3.3 配置 UI 是硬编码的

当前 `src/components/preview/PreviewToolbar.vue` 直接把所有控件写死：

- 字体
- 主题色
- H1/H2/H3 字号
- 正文字号
- 日期字号/字重
- 行高
- 段距
- 页边距

问题：

- 模板无法自定义自己的配置项
- 模板无法隐藏不适用配置
- 模板无法新增专属配置
- 以后要做用户模板编辑器时无法复用

### 3.4 渲染运行时存在大量硬编码

当前 `src/components/PreviewPane.vue` 中包含大量运行时注入：

- `buildPreviewStyles` 生成公共 CSS
- 证件照 DOM 结构固定
- 标题、日期、联系信息的运行时增强规则固定
- 页面脚注和分页样式固定

问题：

- 模板自由度被运行时 CSS 抢走
- 某些布局能力无法完全下放给模板
- 预览层承担了过多模板职责

### 3.5 后端模板加载与打包粒度过粗

当前 Rust 侧模板扫描只识别 `.css` 文件，`tauri.conf.json` 也只打包 `src/assets/templates/*.css`。

问题：

- 无法支持模板目录结构
- 无法支持 manifest
- 无法支持模板资源文件
- 无法支持模板预设、schema、示例或缩略图

## 4. 目标架构

## 4.1 模板包结构

将模板从单文件升级为目录包。

推荐目录结构：

```text
src/assets/templates/
  modern/
    template.json
    style.css
    assets/
      preview.png
      icon-phone.svg
  business/
    template.json
    style.css
    assets/
```

用户模板目录与内置模板保持一致：

```text
AppData/templates/
  modern/
    template.json
    style.css
    assets/
```

### 4.1.1 模板包内文件职责

- `template.json`: 模板元信息、默认值、配置 schema、能力声明
- `style.css`: 模板主样式入口
- `assets/`: 模板依赖图片、SVG、字体或其他静态资源

### 4.1.2 目录优于单文件的原因

- 更符合未来模板扩展方向
- 元信息、样式和资源不会耦在一份 CSS 注释里
- 用户模板的覆盖行为更清晰
- 打包和导出时可完整复制模板资产

## 4.2 模板 manifest 设计

推荐 `template.json` 结构如下：

```json
{
  "id": "modern",
  "name": "现代简约",
  "version": "1.0.0",
  "entryCss": "style.css",
  "description": "适合社招和校园招聘的双语现代风格简历模板",
  "defaults": {
    "themeColor": "#4c49cc",
    "fontFamily": "\"Microsoft YaHei\", \"PingFang SC\", sans-serif",
    "fontSize": 14,
    "lineHeight": 1.5,
    "paragraphSpacing": 8,
    "h1Size": 32,
    "h2Size": 18,
    "h3Size": 16,
    "dateSize": 14,
    "dateWeight": "700",
    "marginV": 10,
    "marginH": 12,
    "contactMode": "icon"
  },
  "editorSchema": [
    {
      "key": "themeColor",
      "type": "color",
      "label": "主题色"
    },
    {
      "key": "fontFamily",
      "type": "select",
      "label": "字体",
      "options": [
        {
          "label": "苹方",
          "value": "\"PingFang SC\", \"Microsoft YaHei\", sans-serif"
        },
        {
          "label": "微软雅黑",
          "value": "\"Microsoft YaHei\", \"PingFang SC\", sans-serif"
        }
      ]
    },
    {
      "key": "fontSize",
      "type": "number",
      "label": "正文字号",
      "min": 11,
      "max": 16,
      "step": 1
    }
  ],
  "features": {
    "photo": true,
    "pageFooter": true,
    "contactModes": ["text", "icon"]
  }
}
```

### 4.2.1 必填字段

- `id`
- `name`
- `version`
- `entryCss`
- `defaults`
- `editorSchema`

### 4.2.2 可选字段

- `description`
- `features`
- `thumbnail`
- `author`
- `tags`
- `layout`

### 4.2.3 约束要求

- `id` 在整个模板集合内唯一
- `defaults` 必须覆盖 `editorSchema` 中出现的 key
- `editorSchema.key` 不允许重复
- 配置项类型必须可序列化

## 4.3 前端模板数据结构改造

### 4.3.1 新的数据结构

建议替换当前固定 `ResumeStyle` 方案，改为通用模板值模型：

```ts
type TemplateValue = string | number | boolean

interface TemplateOption {
  label: string
  value: TemplateValue
}

interface TemplateFieldSchema {
  key: string
  type: 'color' | 'number' | 'select' | 'boolean' | 'text'
  label: string
  min?: number
  max?: number
  step?: number
  options?: TemplateOption[]
}

interface ResumeTemplate {
  id: string
  name: string
  version: string
  description?: string
  css: string
  defaults: Record<string, TemplateValue>
  editorSchema: TemplateFieldSchema[]
  features?: Record<string, unknown>
}

interface ResumeRenderProfile {
  templateId: string
  values: Record<string, TemplateValue>
  photoPath?: string
}
```

### 4.3.2 改造收益

- 模板配置变成纯数据
- 新模板新增配置项无需改 store 类型
- 用户模板和内置模板共用同一结构
- 后续模板编辑器只操作 JSON，不再碰业务代码

## 4.4 渲染契约改造

### 4.4.1 稳定语义化 DOM

核心渲染器继续负责 Markdown -> HTML，但需要收敛出稳定的语义类名和结构，例如：

- `.resume-document`
- `.personal-header`
- `.job-intention`
- `.contact-info`
- `.contact-info--text`
- `.contact-info--icon`
- `.contact-info-item`
- `.experience-line`
- `.experience-title`
- `.experience-date`
- `.resume-photo-wrapper`
- `.manual-page-break`

目标是：

- 核心程序只负责输出语义结构
- 模板只负责消费这些语义结构

### 4.4.2 用 CSS 变量替代反向解析

由核心程序统一注入模板变量：

```css
.resume-document {
  --tpl-theme-color: #4c49cc;
  --tpl-font-family: "Microsoft YaHei", "PingFang SC", sans-serif;
  --tpl-font-size: 14px;
  --tpl-line-height: 1.5;
  --tpl-paragraph-spacing: 8px;
  --tpl-h1-size: 32px;
  --tpl-h2-size: 18px;
  --tpl-h3-size: 16px;
  --tpl-date-size: 14px;
  --tpl-date-weight: 700;
  --tpl-page-margin-v: 10mm;
  --tpl-page-margin-h: 12mm;
}
```

模板 CSS 通过变量消费配置：

```css
@page {
  margin: var(--tpl-page-margin-v) var(--tpl-page-margin-h);
}

.resume-document {
  font-family: var(--tpl-font-family);
  font-size: var(--tpl-font-size);
  line-height: var(--tpl-line-height);
}

.resume-document h2 {
  color: var(--tpl-theme-color);
}
```

这样以后新增字段的流程变成：

1. 模板 manifest 新增 `defaults + editorSchema`
2. 通用运行时把值转为 CSS 变量
3. 模板 CSS 使用变量

不再需要写新的字段解析器。

### 4.4.3 公共运行时与模板运行时职责拆分

保留公共运行时，只承担“所有模板都需要”的基础职责：

- 字体资源注入
- Paged.js 预览骨架
- 分页容器通用样式
- 证件照基础能力
- 手动分页能力
- 共享语义增强逻辑

模板运行时只负责：

- 模板 CSS
- 模板变量消费
- 模板资源引用

不允许公共运行时继续替模板决定主题色、字号、标题色等具体视觉表达。

## 4.5 通用配置 UI

### 4.5.1 目标

把 `PreviewToolbar.vue` 从“写死控件集合”重构为“根据 `editorSchema` 自动渲染控件”。

### 4.5.2 通用控件类型

首期支持：

- `color`
- `number`
- `select`
- `boolean`
- `text`

必要时二期再支持：

- `slider`
- `group`
- `section`
- `radio`
- `font`

### 4.5.3 生成式 UI 规则

- 读取当前模板的 `editorSchema`
- 为每个字段生成对应控件
- 值写入当前文件的 `renderProfile.values`
- 变更后触发预览更新与持久化

### 4.5.4 设计要求

- 保持与现有 Soft Minimalist 风格一致
- 不新增模板专用组件分支逻辑
- 同类控件尽量复用一个渲染器

## 4.6 后端模板服务改造

### 4.6.1 命令改造目标

当前命令只支持 `list_templates` 和 `save_template(id, css)`，未来需要升级为模板包读写。

建议新增或重构为：

- `list_templates`
- `read_template_package`
- `save_template_package`
- `delete_template_package`
- `copy_builtin_template_to_user_space`

### 4.6.2 `list_templates` 新职责

- 扫描内置模板目录
- 扫描用户模板目录
- 读取每个模板目录中的 `template.json`
- 读取 `entryCss`
- 按模板 id 合并用户覆盖
- 返回完整模板包信息

### 4.6.3 覆盖规则

- 用户模板与内置模板同 id 时，用户模板优先
- 用户模板必须是“完整模板包覆盖”，不做半覆盖
- 减少复杂合并，避免维护成本增加

### 4.6.4 打包调整

`tauri.conf.json` 需要从打包单个 `.css` 改成打包整个模板目录，例如：

- 打包 `src/assets/templates/**`
- 保证 `template.json`、`style.css`、`assets` 一并进入资源目录

## 4.7 用户自定义模板能力规划

### 4.7.1 功能边界

用户自定义模板的目标不是“写脚本”，而是：

- 复制现有模板包
- 修改模板元信息
- 修改 schema
- 修改默认值
- 编辑 CSS
- 保存为用户模板

### 4.7.2 用户模板创建流程

推荐流程：

1. 用户选择一个基础模板
2. 后端复制模板包到用户目录
3. 为其分配新的模板 id
4. 用户在 UI 中编辑模板名称、字段、默认值和 CSS
5. 保存为独立模板包
6. 模板列表自动刷新

### 4.7.3 首期不做的能力

为了控制复杂度，首期不要支持：

- 模板内执行任意 JS
- 模板自定义 Vue 组件
- 模板自定义 Markdown 解析器
- 模板自定义导出引擎

## 5. 分阶段改造计划

## 阶段 0：准备与冻结

### 目标

在正式改造前整理边界，避免过程中出现“做一半又加字段”的反复返工。

### 任务

- 盘点现有模板能力清单
- 盘点现有 `ResumeStyle` 字段与对应 UI 控件
- 盘点预览与导出链路中的模板相关注入
- 确认首期要支持的 schema 字段类型
- 确认模板资源目录结构

### 输出

- 模板字段映射表
- 现有模板兼容清单
- 新模板 manifest 草案

### 验收标准

- 新旧模型映射关系清楚
- 首期字段范围冻结
- 不再新增临时字段

## 阶段 1：模板包基础设施

### 目标

先让系统支持目录化模板包。

### 主要改动点

- `src-tauri/src/lib.rs`
- `src-tauri/tauri.conf.json`
- `src/assets/templates/*`

### 任务

- 将内置模板改成目录结构
- 在 Rust 侧实现模板目录扫描
- 实现 `template.json` 读取与反序列化
- 按 `entryCss` 读取 CSS
- 返回完整模板包给前端
- 保留用户模板覆盖逻辑
- 打包配置改为整个模板目录

### 交付结果

- 前端可拿到完整模板包对象
- 新增模板目录后可自动出现在列表中

### 验收标准

- 现有内置模板都能正常显示
- 用户模板覆盖仍然有效
- 打包后的程序能读取完整模板包

## 阶段 2：前端数据模型重构

### 目标

移除固定 `ResumeStyle` 的中心地位，改成通用 `values` 模型。

### 主要改动点

- `src/stores/resume.ts`
- `src/utils/templateStyle.ts`

### 任务

- 新增通用模板 schema 类型
- `ResumeTemplate` 改成完整模板模型
- `ResumeRenderProfile.style` 改成 `values`
- 持久化 JSON 结构升级
- 加入兼容迁移逻辑：
  - 老版本 `style` 自动转换为 `values`
  - 缺失字段用 `defaults` 回填
- 删除 CSS 反向解析依赖

### 兼容策略

首期不能直接断掉旧数据，必须支持旧工作区 render state 自动迁移：

- 读取到旧 `style` 时，映射为新字段值
- 写回时统一写新结构

### 验收标准

- 旧工作区可无感打开
- 切换模板、保存文件、重启应用后配置不丢失
- 无需再从 CSS 解析默认值

## 阶段 3：渲染链路改造

### 目标

让预览和导出从“依赖固定样式字段”改成“依赖模板变量 + 语义化 DOM”。

### 主要改动点

- `src/components/PreviewPane.vue`
- `src/utils/exportStyles.ts`
- `src/components/TopNavBar.vue`
- `src/utils/resumeParser.ts`

### 任务

- 实现 `values -> CSS variables` 的统一注入函数
- 收缩 `buildPreviewStyles`，只保留公共基础层
- 模板特定样式全部归位到模板 CSS
- 确保导出 HTML 与预览使用同一套变量结果
- 复核联系信息、证件照、经验时间等语义增强是否足够稳定

### 验收标准

- 同一模板在预览与导出 PDF 中视觉一致
- 模板 CSS 能真正决定大多数视觉表现
- 公共运行时不再硬编码具体模板主题风格

## 阶段 4：配置 UI 通用化

### 目标

把模板配置面板改成 schema-driven。

### 主要改动点

- `src/components/preview/PreviewToolbar.vue`
- 可新增：`src/components/template-config/*`

### 任务

- 抽离通用字段渲染器
- 根据 `editorSchema` 动态渲染控件
- 把现有硬编码控件映射成 schema
- 支持字段校验、默认值回退和即时预览

### 设计要求

- 组件结构尽量通用
- 不为单个模板写条件分支
- 不允许在 UI 层重新硬编码字段名称

### 验收标准

- 现有模板配置能完整通过 schema 渲染
- 新增模板字段无需改 UI 代码
- 切换模板时控件自动切换

## 阶段 5：模板保存与用户模板创建

### 目标

支持用户模板包持久化，为后续模板编辑器打基础。

### 主要改动点

- `src-tauri/src/lib.rs`
- `src/stores/resume.ts`
- 可新增模板管理界面

### 任务

- 将“保存模板”从 `save_template(id, css)` 升级为保存完整模板包
- 支持复制内置模板到用户空间
- 支持用户模板重命名、另存为、覆盖保存
- 支持删除用户模板

### 验收标准

- 用户可基于内置模板创建自己的模板
- 用户模板重启后仍可被读取
- 同 id 覆盖规则稳定且可预测

## 阶段 6：模板自定义功能

### 目标

在前面基础稳定后，再做真正面向用户的模板编辑能力。

### 任务

- 模板 manifest 编辑界面
- 模板字段 schema 编辑界面
- CSS 编辑器
- 模板预览缩略图
- 模板导入导出

### 验收标准

- 普通用户不修改源码也能创建模板
- 导入导出模板包后可在另一台机器复用

## 6. 文件级改造建议

## 6.1 后端

### `src-tauri/src/lib.rs`

职责调整：

## 7. 兼容与迁移策略

## 7.1 旧模板兼容

首期可以允许老模板通过自动包装方式兼容：

- 发现旧模板只有 CSS 文件时
- 自动生成临时 manifest
- 使用预设 schema 映射一组标准字段

但这只是过渡方案，不建议长期保留。

## 7.2 旧 render state 兼容

现有 `.max-md2cv.render-state.json` 中的：

- `templateId`
- `style`
- `photoPath`

需要迁移为：

- `templateId`
- `values`
- `photoPath`

迁移策略：

1. 读取时判断是否存在旧 `style`
2. 若存在，则映射到新 `values`
3. 运行期间统一使用新结构
4. 下一次保存时写回新结构

## 7.3 分阶段发布策略

建议分两次发布：

- 第一次发布：模板包 + 新 store + 新渲染链路，先不开放用户模板编辑器
- 第二次发布：模板管理与用户模板创建

这样可以降低一次性改动过大导致的不稳定性。

## 8. 风险与应对

## 8.1 风险：预览和导出不一致

原因：

- 预览和导出使用不同样式注入路径

应对：

- 抽出统一模板运行时样式构建函数
- 所有变量注入和模板 CSS 拼装只走一条路径

## 8.2 风险：旧数据丢失

原因：

- render state 结构升级

应对：

- 保留自动迁移
- 在开发阶段准备多组旧数据样本回归

## 8.3 风险：schema-driven UI 过早复杂化

原因：

- 一次支持太多字段类型

应对：

- 首期只支持 5 种字段类型
- 先覆盖当前已有能力
- 后续再增量扩展

## 8.4 风险：模板能力边界失控

原因：

- 试图在模板中引入任意脚本或自定义渲染器

应对：

- 明确禁止可执行模板
- 只允许声明式元数据和 CSS

## 9. 测试与验收清单

## 9.1 功能验收

- 新增一个模板目录后，无需改代码即可出现在模板列表
- 模板切换后，预览正确更新
- 模板配置修改后，预览即时更新
- 模板配置可持久化到工作区 render state
- 导出 PDF 与预览视觉一致
- 用户模板可覆盖同 id 内置模板

## 9.2 兼容验收

- 旧工作区 render state 可自动迁移
- 现有内置模板表现不回退
- 用户原有 CSS 模板在过渡期可继续使用

## 9.3 稳定性验收

- 连续切换模板不会报错
- 资源缺失时能给出明确错误
- 非法 manifest 不会导致整个模板列表崩溃

## 10. 推荐实施顺序

建议的最优顺序如下：

1. 模板包目录和 manifest 落地
2. Rust 模板扫描与返回结构升级
3. 前端模板模型升级
4. render state 迁移逻辑
5. CSS variables 统一注入
6. 预览/导出链路统一
7. schema-driven 配置面板
8. 用户模板保存与管理
9. 模板自定义功能

## 11. 首期实施范围建议

如果希望尽快落地，建议首期只做这些：

- 模板包目录化
- manifest 加载
- `values` 取代 `ResumeStyle`
- CSS variables 注入
- schema-driven 基础配置 UI
- 旧数据迁移

先不要做：

- 可视化模板编辑器
- 模板导入导出
- 模板字段设计器

这样可以先把底座搭稳，再在其上做用户自定义模板功能。

## 12. 最终预期效果

改造完成后，新增模板的流程应当收敛为：

1. 新建一个模板目录
2. 写 `template.json`
3. 写 `style.css`
4. 放入资源文件
5. 启动应用或刷新模板列表

不需要再修改：

- Rust 模板列表代码
- 前端模板类型
- 预览工具栏
- 样式解析器
- 导出逻辑

这才是后续做“用户可配置模板、自定义模板市场、模板导入导出”的正确底座。
